# 001 Architecture 架构规范

## 目的

定义 Fauplay 的系统架构边界与分层职责，确保后续能力扩展在同一架构约束下演进。

## 关键术语 (Terminology)

- 浏览器前端（Web App）
- 本地能力网关（Local Capability Gateway / MCP Host）
- 能力插件（MCP Plugins）
- 工具发现（Tool Discovery）
- 工具调用（Tool Call）
- 优雅降级（Graceful Degradation）

## 范围

范围内：

- Web 应用、本地能力网关、MCP 插件三层架构边界。
- 运行时依赖方向与调用链路约束。
- 网关可选化与降级策略约束。
- 同源远程只读 HTTP 数据面的发布边界与职责归属。
- 同源远程媒体会话、原生媒体请求与服务端分段读取能力的职责归属。
- 新能力接入的架构入口规则。

范围外：

- HTTP/JSON-RPC 字段与错误码细节（归属 `002-contracts`）。
- 跨专题安全基线、会话治理与 secret/logging 最小化规则（归属 `006-security`）。
- UI 命名、布局分区、交互细则（归属 `003-ui-ux`）。
- 单个功能专题的实现步骤（归属 `100+` 主题）。

## 架构分层

Fauplay 采用三层架构：

1. Web App（浏览器前端）
2. Local Capability Gateway（本地网关 / MCP Host）
3. MCP Plugins（能力插件层）

调用方向固定为：

`Web App -> Gateway -> Plugins`

补充约束：

- `Web App -> Gateway` 既可承载本机回环 MCP/HTTP 增强链路，也可承载同源远程只读 HTTP 数据面。
- 远程只读 HTTP 数据面不改变插件层依赖方向；其内部仍由 Gateway 统一编排数据读取与能力裁剪。

禁止反向依赖：

- 插件层不得依赖前端 UI。
- 网关层不得直接耦合具体页面组件。

## 分层职责

### Web App

职责：

- 目录授权、文件浏览、筛选排序、预览与交互。
- 访问提供者切换与统一消费（如 `local-browser`、`remote-readonly`）。
- 工具发现（Tool Discovery）与工具调用（Tool Call）发起（通过网关客户端）。
- 网关不可用时保持核心浏览链路可用。

约束：

- 组件层不直接耦合系统调用实现细节。
- 系统能力调用应通过统一分发入口。

### Local Capability Gateway

职责：

- 作为本地能力入口，承载工具发现与工具调用路由。
- 作为同源只读数据入口，可选承载局域网远程浏览所需的受限只读 HTTP 路由。
- 作为同源远程会话入口，可选承载远程登录交换、会话管理、媒体分段读取与缩略图派生资源。
- 承载插件生命周期管理（初始化、调用、异常隔离）。
- 输出统一响应结构与错误语义（细节见 `002-contracts`）。

约束：

- 默认监听回环地址，不直接暴露公网。
- 对 LAN 发布时，必须通过外部反向代理终止 HTTPS，且仅公开经显式设计的最小只读远程入口；具体安全基线见 `006-security`。
- 插件加载走白名单策略。

### MCP Plugins

职责：

- 实现具体系统能力（如系统文件定位、系统默认打开等）。
- 对外只暴露工具描述与工具调用能力。

约束：

- 不感知前端布局与组件状态。
- 输入输出遵循网关宿主约束，不绕过网关直接对前端暴露接口。

## 运行与降级策略

1. 核心浏览体验默认不依赖网关在线状态。
2. 网关离线时，前端进入优雅降级（Graceful Degradation）模式：
   - 不阻断目录浏览、筛选、预览。
   - 仅隐藏或禁用系统增强动作。
3. 网关恢复后，前端可通过能力刷新恢复可用动作。
4. 远程只读模式属于显式接入的访问提供者，不得通过暴露本机回环 Gateway 或 legacy 绝对路径接口直接替代。
5. loopback 发布面、LAN 发布面与 dev-only 联调面必须视为不同暴露层级；最小暴露面与安全默认原则由 [`../006-security/spec.md`](../006-security/spec.md) 统一约束。

## 架构演进规则

1. 新系统能力优先按“插件工具”方式接入，不在组件内硬编码平台逻辑。
2. 新增跨层能力前，先明确所属层级与边界，再推进实现。
3. 涉及接口或错误语义变更时，先更新 `002-contracts`，再实现代码变更。
4. 与 `000-foundation` 冲突时，先更新 foundation 基线，再推进架构调整。
5. 涉及信任边界、最小暴露面、会话治理或 secret/logging 最小化的变更时，先更新 `006-security`，再推进架构或功能实现。

## 非目标

1. 不定义具体 endpoint、请求/响应字段。
2. 不定义组件目录结构与命名契约。
3. 不定义任务拆解、排期与里程碑。
