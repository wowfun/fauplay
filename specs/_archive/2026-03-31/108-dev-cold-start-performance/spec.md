# 108 Dev Cold Start Performance 冷启动性能专题规范

> Archived on 2026-03-31. Kept for historical reference.

## 1. 目的 (Purpose)

定义 Fauplay 在开发模式下“服务重启后首次刷新进入开始页慢”的专项性能契约，确保开始页优先可见、工作区重模块延后加载，并为首次进入工作区提供可感知反馈。

## 2. 问题定义与现状证据 (Problem & Evidence)

问题定义：

1. 执行 `npm run dev` 后，用户点击浏览器刷新，开始页出现明显延迟。
2. 控制台中 React DevTools 提示出现较晚，且开始页通常在该提示后才可见。

现状证据（采样日期：2026-03-08）：

1. 冷启动首次请求 `/`：`ttfb` 约 `1.60s`。
2. 同轮中 `main.tsx` 转换耗时约 `1846ms`，`index.css` 约 `660ms`。
3. 开始页场景下仍触发部分工作区重模块转换（约 `200ms-540ms`），造成首屏等待放大。

## 3. 范围与非目标 (In Scope / Out of Scope)

范围内：

1. 开始页路径的加载链路收敛与首屏可见性优化。
2. 工作区重模块的延后加载（按需加载）。
3. 首次进入工作区时的加载占位反馈。
4. 稳定优先的开发依赖预热策略。

范围外：

1. 生产构建产物体积优化与线上运行时性能改造。
2. 非冷启动场景的通用性能重构。
3. 业务功能语义调整（地址栏、预览逻辑等行为保持不变）。

## 4. 专项目标 (Goals)

1. 用户刷新后，开始页应在最小依赖路径下尽快可见，不等待工作区重模块完成转换。
2. 未选目录前，不主动加载工作区重模块与其副作用链路。
3. 首次进入工作区时，必须有明确加载占位，避免“无反馈等待”。
4. 依赖优化优先采用稳定策略，避免引入实验参数导致不确定行为。
5. 目录首屏渲染必须优先于文件元数据补全，避免因全量 `getFile()` 阻塞工作区可见性。
6. 工作区首屏（工具栏 + 网格 + 状态栏）必须独立于插件/预览模块冷编译，保证进入目录后先可交互。
7. 点击文件后，预览媒体视口必须优先可见；插件运行时链路不得阻塞预览首帧。

## 5. 实现约束 (Implementation Constraints)

1. 开始页与工作区入口应拆分，工作区模块按需加载。
2. `useGatewayCapabilities` 等工作区副作用应后置到工作区装配层，避免开始页阶段触发。
3. 开发服务器优化优先使用 `server.warmup.clientFiles` 预热关键入口（开始页 + 工作区核心渲染链路，不含插件/预览重模块）。
4. 本专题默认不引入实验性依赖优化参数（如 `holdUntilCrawlEnd`）。
5. 目录扫描默认采用“快速列表”策略：首轮仅产出渲染所需最小字段，文件大小/修改时间等元数据允许延后获取。
6. 视频缩略图链路必须具备失败快速返回与资源清理机制，避免长时间挂起或重复 `blob` 资源错误。
7. 插件运行时与预览工具链路必须按需懒加载；无 gateway 或未打开预览时，不应成为工作区首屏前置依赖。
8. gateway 能力探测应异步执行，不得阻塞工作区核心视图首屏展示。
9. 预览面板内部需分层加载：媒体渲染主路径与插件运行时主路径解耦，插件链路仅在存在可用工具时再加载。

## 6. 功能需求 (FR)

1. `FR-DCSP-01` 未选目录前，开始页渲染不得被工作区重模块加载阻塞。
2. `FR-DCSP-02` 工作区模块必须通过按需加载机制在进入工作区时才加载。
3. `FR-DCSP-03` 首次进入工作区必须展示加载占位直到工作区模块可用。
4. `FR-DCSP-04` Dev 依赖优化必须采用稳定优先策略（`warmup` 优先，禁用实验参数作为默认方案）。
5. `FR-DCSP-05` 本专题不得引入运行时代码公共 API 变更。
6. `FR-DCSP-06` 目录首轮读取不得为每个媒体文件强制读取完整 `File` 对象。
7. `FR-DCSP-07` 视频缩略图任务在不可解码或异常场景下必须在可控时间内失败并释放临时资源。
8. `FR-DCSP-08` 工作区基础视图渲染不得依赖 `WorkspacePluginHost`、`FilePreviewPanel`、`FileLightboxModal` 的模块加载完成。
9. `FR-DCSP-09` 预览面板中媒体渲染主路径不得依赖 `usePluginRuntime` 与插件结果面板组件的模块加载完成。
10. `FR-DCSP-10` 未启动 gateway 且工具列表为空时，首次打开预览不得主动请求插件运行时模块链路。

## 7. 验收标准 (AC)

1. `AC-DCSP-01` 冷启动后首次刷新时，开始页可先于工作区重模块可用状态出现。
2. `AC-DCSP-02` 未选目录前的模块请求中，不应出现工作区懒加载 chunk 的主动拉取。
3. `AC-DCSP-03` 首次进入工作区时可见加载占位，加载完成后正常进入工作区界面。
4. `AC-DCSP-04` 启用 `warmup` 后，冷启动首刷等待时间相较基线有可量化改进，且无实验参数依赖。
5. `AC-DCSP-05` 地址栏导航、预览、插件运行等既有功能行为无回归。
6. `AC-DCSP-06` 进入工作区时可在无缩略图前置完成的情况下先展示文件网格与基础信息。
7. `AC-DCSP-07` 不支持的视频文件缩略图失败应快速返回，且控制台不出现持续重复的 `blob:* net::ERR_FILE_NOT_FOUND` 噪声。
8. `AC-DCSP-08` 在未启动 gateway 场景下，进入工作区时网络瀑布中不应出现插件/预览链路模块作为首屏阻塞项。
9. `AC-DCSP-09` 点击文件打开预览时，媒体区域可在插件运行时模块未完成转换前先出现加载态或内容态。
10. `AC-DCSP-10` 未启动 gateway 且 `previewActionTools` 为空时，预览首开网络瀑布不应出现 `src/features/plugin-runtime/*` 请求。

## 8. 失败与降级行为 (Failure & Degradation)

1. 若工作区懒加载失败，系统应提供可见错误提示并允许用户重试。
2. 若 `warmup` 不可用或失效，系统应降级为默认 dev 行为，不影响功能正确性。
3. 任何性能优化失败不得阻断开始页基本可用路径。

## 9. 测试与验收场景 (Test Scenarios)

1. 冷启动首刷场景：`npm run dev` 后首次刷新，记录开始页可见时间与请求时序。
2. 路由与请求场景：未选目录前抓取网络请求，验证工作区模块未被提前请求。
3. 首次进入工作区场景：验证加载占位可见、消失时机正确、最终界面正常。
4. 功能回归场景：目录选择、地址栏导航、预览与插件流程保持既有语义。
5. 预览首开场景（gateway 关闭）：点击文件后记录请求时序，验证媒体视口先出现，且无插件运行时链路请求。

## 10. 公共接口与类型影响 (Public Interfaces & Types)

1. 本专题不新增运行时公共 API 或协议字段。
2. 本专题新增文档级接口约束：`FR-DCSP-*` 与 `AC-DCSP-*` 编号、`P50/P95` 指标口径、五项治理信息集。

## 11. 默认值与一致性约束 (Defaults & Consistency)

1. 专题编号固定为 `108-dev-cold-start-performance`。
2. 稳定优先默认策略固定为：`warmup` 开启，实验参数关闭。
3. 本专题必须引用 `004-performance-governance` 作为治理上游规范。
