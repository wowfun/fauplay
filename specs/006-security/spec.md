# 006 Security 基础安全规范

## 1. 目的 (Purpose)

定义 Fauplay 的统一安全基线，作为 `000-099` 基础主题中的跨专题上游约束，确保：

1. Web App、本地能力网关（Gateway）、局域网远程只读入口与浏览器本地状态在同一安全边界下演进。
2. 安全约束以“默认安全（Security by Default）+ 最小权限（Least Privilege）+ 最小暴露面（Minimum Exposure Surface）”为基础，而不是依赖调用方自觉遵守。
3. 功能专题在涉及鉴权、路径、暴露面、secret、日志或浏览器持久化时，可复用统一上游规则，而不是各自发明语义。
4. 本规范只定义安全基线与治理要求，不直接替代 `002-contracts` 的协议契约，也不直接承接单一功能整改步骤。

## 2. 关键术语 (Terminology)

- 威胁模型（Threat Model）
- 信任边界（Trust Boundary）
- Secret（密钥 / 凭证）
- 会话（Session）
- 能力暴露面（Capability Exposure Surface）
- Allowlist（白名单）
- 最小权限（Least Privilege）
- 默认安全（Security by Default）
- 资源滥用（Resource Abuse）

术语口径：

1. `Threat Model`：系统在当前主题下默认需要防御的输入来源、调用路径与误配置风险集合。
2. `Trust Boundary`：不同组件、运行环境或网络入口之间默认不互信的边界，例如浏览器输入、loopback、本地文件系统、LAN 反向代理入口。
3. `Secret`：任何可用于建立身份、换取会话或访问受限能力的高敏感度值，例如 token、cookie、进程环境凭证。
4. `Session`：登录或初始化后由系统维护的运行态授权状态，不等同于一次性登录凭证。
5. `Capability Exposure Surface`：对外可访问的 HTTP/MCP/文件/浏览器持久化入口集合。
6. `Resource Abuse`：合法或非法请求通过放大计算、IO、内存、并发或递归遍历成本而造成的可用性风险。

## 3. 范围与非目标 (In Scope / Out of Scope)

范围内：

1. Fauplay 的统一威胁模型与信任边界约束。
2. 鉴权、会话、路径授权、资源授权与派生资源授权的上游规则。
3. 最小暴露面、LAN 发布边界、dev-only 能力标识与调试入口隔离原则。
4. Secret、日志与浏览器本地持久化的最小化约束。
5. 跨专题安全治理与文档引用规则。

范围外：

1. 具体 HTTP schema、错误码字段与 JSON wire shape（归属 `002-contracts`）。
2. 单一功能专题的具体整改步骤、补丁设计与代码实现。
3. 特定反向代理配置模板，或强制绑定某一种部署产品。
4. 具体浏览器安全头实现细节（如 `CSP`、`HSTS`、`frame-ancestors`）的首期强制规范。

## 4. 核心安全域 (Core Security Domains)

### 4.1 威胁模型与信任边界

1. 浏览器输入、HTTP 请求参数、Header、Cookie、MCP 参数与浏览器本地持久化内容都必须视为不可信输入。
2. Web App、Gateway、本地文件系统、插件进程与 LAN 发布入口必须视为不同信任边界；跨边界访问必须通过显式契约完成。
3. `127.0.0.1` / loopback 发布面与同源 LAN 远程发布面必须视为两种不同安全级别，不得混用“本机可访问”与“局域网可访问”的默认前提。
4. Gateway 可复用内部数据层与能力层，但远程公开入口不得因此自动继承 legacy 或调试入口的暴露资格。
5. 功能专题若改变攻击面、发布面或信任边界，必须在其稳定规范中显式声明变化。

### 4.2 鉴权与会话

1. Secret 不得进入公开 JSON 配置、公开响应、浏览器长期存储或不必要日志。
2. 一次性登录凭证与运行态会话必须分离；系统不得要求客户端在运行期持续重复提交高敏感度登录凭证，除非该专题明确声明原因。
3. 会话机制必须定义至少四类语义：
   - 创建（Create）
   - 失效（Expire / Invalidate）
   - 清理（Cleanup）
   - 异常恢复（Recovery after invalid session）
4. 鉴权失败响应必须统一、最小化，不得泄露“用户名不存在 / token 长度不对 / root 不存在”等额外探测信息。
5. 面向网络入口的认证流程必须考虑暴力尝试与资源滥用风险，不得把“凭证强度由用户自己保证”作为唯一防线。
6. 若系统提供 remember-device / trusted-device 一类长效凭证，它必须与短期 session 分离、显式 opt-in、支持服务端撤销，并且服务端持久化层只允许保存不可逆摘要或等价最小化表示。

### 4.3 路径与资源授权

1. 公开输入不得直接接受绝对路径作为跨边界资源访问主语义，除非该能力被明确界定为本机受限调试入口。
2. 路径类输入必须做归一化（Path Normalization）、越界阻断与作用域校验，不得依赖前端自觉传入正确路径。
3. 当系统以 allowlist / root / scope 约束资源可见性时，所有公开资源访问都必须经过同一授权链路。
4. 衍生资源（例如缩略图、裁切图、预览缓存、流式片段）不得绕过原始资源的授权边界。
5. 当资源通过标识符（如 `assetId`、`faceId`、`jobId`）访问时，系统仍必须验证该标识符对应对象是否落在当前允许的授权范围内。

### 4.4 暴露面与发布边界

1. 默认仅开放最小必要入口；任何额外入口都必须先被明确归类为：
   - 生产发布面
   - 本机调试面
   - dev-only 联调面
2. legacy 接口、内部调试入口、管理入口与通用宿主入口默认不得对 LAN 暴露，除非专题规范显式声明并给出安全边界。
3. dev-only 代理、调试代理或 HTTPS 联调能力必须显式标记，不得被视为生产默认发布能力的放宽。
4. 功能专题若依赖“正确反向代理配置”作为安全前提，必须同时明确“错误暴露时会失守的边界”，避免误把部署约定当成内部强约束。
5. 本机 admin 面与 LAN 远程面必须严格分离；loopback-only 管理接口不得复用 `/v1/remote/*` 公开数据面，也不得依赖“前端不显示入口”作为唯一隔离手段。
6. 若 loopback-only 管理接口因本机同步需要接收 `absolutePath` 或其他高敏路径输入，该能力也必须限制在 loopback/admin 暴露面内，不得复制到 LAN 远程公开面。

### 4.5 Secret / 日志 / 本地持久化最小化

1. token、cookie、绝对路径、敏感查询参数与高敏感度 Header 不得进入不必要日志、错误消息或前端持久化状态。
2. 浏览器本地收藏、历史、最近访问、布局状态与其他本地持久化数据必须按服务边界或作用域隔离，避免跨服务串用。
3. 本地持久化只允许保存实现体验所必需的最小信息，不得把可重新获取的高敏感度授权状态长期持久化。
4. 错误诊断信息必须在“可排障”与“不过度泄露”之间保持最小必要原则；默认优先输出抽象状态而不是敏感原值。
5. 长效设备凭证若需要服务端持久化，持久化内容必须最小化，且不得以明文形式保存可直接复用的设备 token。
6. 设备管理或会话管理类管理接口返回的数据必须最小化，不得暴露 cookie 原值、token 原值、完整 `User-Agent` 原文、完整来源 IP 或其他非必要敏感字段。
7. 服务端私有状态文件（如自动发布 roots、远程共享收藏、remember-device 状态）不得被误归类为浏览器本地状态或公开 JSON 配置；公开响应只允许返回最小 DTO。

## 5. 治理契约 (Governance Contract)

1. 新能力若引入新的网络入口、资源标识、路径访问语义、登录态或浏览器持久化状态，必须在对应专题中显式说明其信任边界与攻击面变化。
2. 公开资源的授权链路必须同时覆盖主资源与派生资源，不得只保护列表接口而放任衍生读取入口绕过授权。
3. 会话机制必须有生命周期与异常恢复语义，不得只定义“如何建立会话”而不定义“何时失效、如何清理、失效后如何回退”。
4. 面向网络入口的认证与查询链路必须考虑资源滥用风险；当请求成本可被显著放大时，专题必须定义限制策略或降级策略。
5. 任何 dev-only 能力、调试入口或联调代理都必须显式标记，不得隐式转为生产默认。
6. 功能专题若涉及鉴权、路径、暴露面、secret、日志或浏览器持久化，必须显式引用本规范作为上游基线。

## 6. 文档级接口 (Public Interfaces)

1. 安全基础专题编号固定为：`006-security`。
2. 本规范的需求编号前缀固定为：`FR-SEC-*`。
3. 本规范的验收编号前缀固定为：`AC-SEC-*`。
4. 新功能专题若涉及以下任一安全域，必须显式引用 `006-security`：
   - 鉴权 / 会话
   - 路径 / 资源授权
   - 网络暴露面 / LAN 发布边界
   - Secret / 日志 / 浏览器持久化

## 7. 功能需求 (FR)

1. `FR-SEC-01` 系统必须将浏览器输入、HTTP 参数、Cookie、Header、MCP 参数与浏览器本地持久化内容统一视为不可信输入。
2. `FR-SEC-02` 系统必须把 loopback、本地文件系统、LAN 发布面、Gateway 与插件进程视为不同信任边界，跨边界访问必须通过显式契约完成。
3. `FR-SEC-03` Secret 不得进入公开 JSON 配置、公开响应、浏览器长期持久化与不必要日志。
4. `FR-SEC-04` 登录凭证与运行态会话必须解耦；会话机制必须定义创建、失效、清理与异常恢复语义。
5. `FR-SEC-05` 鉴权失败响应必须统一、最小化，不得泄露额外探测信息。
6. `FR-SEC-06` 公开路径与资源访问不得依赖前端自觉传参；系统必须执行路径归一化、作用域校验与越界阻断。
7. `FR-SEC-07` 当资源受 allowlist / root / scope 约束时，衍生资源访问不得绕过原始资源授权边界。
8. `FR-SEC-08` 系统默认只开放最小必要入口；legacy 面、调试面与 dev-only 面必须与生产发布面显式分离。
9. `FR-SEC-09` 面向网络入口的认证、查询与资源读取链路必须考虑暴力尝试与资源滥用风险。
10. `FR-SEC-10` 浏览器本地收藏、历史、最近访问与其他持久化状态必须按服务边界隔离，并保持最小化。
11. `FR-SEC-11` 新功能专题若改变攻击面、信任边界或公开暴露面，必须在稳定规范中显式声明并引用本规范。
12. `FR-SEC-12` 系统若提供 remember-device / trusted-device 一类长效凭证，必须要求显式启用、支持服务端撤销，并且服务端仅能持久化不可逆摘要或等价最小化表示。
13. `FR-SEC-13` 本机管理面与 LAN 远程公开面必须使用不同暴露面；loopback-only 管理接口不得被远程公开 origin 复用。
14. `FR-SEC-14` 设备管理类接口的响应字段必须最小化，不得泄露 cookie 原值、token 原值、完整 UA 原文或其他非必要敏感元数据。
15. `FR-SEC-15` 若 loopback-only 同步入口需要接收 `absolutePath` 或浏览器私有状态导出的高敏输入，该能力必须固定限制在本机 admin 面，并在跨边界发布时被排除。

## 8. 验收模板 (AC Template)

1. `AC-SEC-01` 任一涉及网络入口的功能专题都能明确指出其信任边界与发布面，不把 loopback 与 LAN 语义混作一层。
2. `AC-SEC-02` 任一涉及鉴权的功能专题都同时定义登录凭证、运行态会话、失效语义与异常恢复语义。
3. `AC-SEC-03` 任一受 root / allowlist / scope 约束的资源入口，都能将同一约束应用到主资源与派生资源。
4. `AC-SEC-04` 任一 dev-only 联调或代理能力都被显式标记，不会被误解为生产默认发布能力。
5. `AC-SEC-05` 任一涉及本地持久化的功能专题都明确说明其持久化范围、隔离维度与敏感信息最小化策略。
6. `AC-SEC-06` 任一 loopback-only 管理接口都能与 LAN 远程入口清晰分离，且其返回数据不包含可直接复用或过度暴露的敏感值。
7. `AC-SEC-07` 任一把浏览器私有状态同步到服务端的 loopback-only 入口，都不会把 `absolutePath`、浏览器私有 `rootId` 或其他高敏输入暴露到远程公开接口与响应中。

## 9. 与上游主题关系 (Relation to 000/001/002)

1. 与 `000-foundation`：本规范细化其“安全与降级基线”，将高层原则收敛为跨专题安全治理要求。
2. 与 `001-architecture`：本规范约束信任边界、发布边界与最小暴露面；架构规范继续负责分层职责与调用方向。
3. 与 `002-contracts`：本规范定义安全上游原则；`002-contracts` 继续负责具体协议层的请求/响应与错误语义契约。

## 10. 默认值与一致性约束 (Defaults & Consistency)

1. 安全基础专题固定为 `006-security`，不与功能专题合并维护。
2. 本规范优先定义安全基线与治理接口，不绑定某个具体功能缺陷的补丁设计。
3. 首期不强制规范具体浏览器安全头实现；若后续需要，可在增量规范或独立专题中扩展。
