# 003 顶部工具栏标签过滤细则 (Top Toolbar Tag Filter Reference)

## 目的

定义顶部工具栏标签过滤的详细交互与状态规则，作为 `003-ui-ux/spec.md` 的引用文档。

## 范围与边界

范围内：

1. 顶部工具栏的“包含标签 / 排除标签”面板行为。
2. `OR/AND`、多选候选、候选计数与面板开关状态。
3. “未标注”特殊项语义。
4. 人物管理虚拟候选项语义。
5. 候选面板内 `source` / `key` 分面规则。
6. 打开面板时的标签快照刷新与面板内无感更新。

范围外：

1. 预览头部标签展示与手动管理（归属 `117-preview-header-tag-management`）。
2. Gateway / SQLite 数据契约（归属 `005-local-data-contracts`）。
3. 文件结果过滤的底层匹配实现细节；本文件仅定义其可见边界。

## 用户可见行为契约

1. 顶部标签过滤位于顶部工具区，用于工作区文件结果的逻辑标签筛选。
2. 过滤 UI 至少包含：
   - 包含标签面板
   - 排除标签面板
   - 包含标签匹配模式 `OR/AND`
3. 顶部标签过滤不提供手动 `all/boolean` 模式切换；过滤模式由 include/exclude 是否为空自动推导。
4. 包含标签与排除标签面板均必须支持多选候选项。
5. 候选项显示默认使用逻辑标签文案 `key: value`；有 `fileCount` 时可显示计数。
6. 同一时刻只允许一个候选面板展开；打开一个面板时必须关闭另一个面板。
7. 候选面板顶部必须提供“全选”与“清空”操作；“全选”仅作用于当前分面后可见的候选项，“清空”作用于当前 include/exclude 已选集合。
8. 顶部标签过滤候选除真实逻辑标签外，还可包含特殊虚拟候选项；首批固定为 `未标注`、`人物管理: 未归属` 与 `人物管理: 误检/忽略`。

## 面板交互与状态契约

1. `source` / `key` 分面状态属于面板级临时状态，不写入全局过滤状态，不持久化到本地存储。
2. 面板首次打开时，`source` 与 `key` 分面默认值均为“全部”。
3. 关闭当前面板、重新打开面板，或从 include 面板切换到 exclude 面板时，分面状态都必须重置为“全部”。
4. 已选中的 include/exclude 标签集合属于工作区过滤状态，面板关闭后不得丢失。
5. 已选中的 include/exclude 标签集合与 `OR/AND` 匹配模式必须按 `rootId` 持久化，并在同一 root 刷新或重进后恢复。
6. 恢复持久化标签过滤时，若某些已保存 `tagKey` 当前已不在候选列表中，系统仍必须保留这些 `tagKey`，不得自动裁剪。
7. `OR/AND` 只影响 include 标签对文件结果的匹配方式，不影响候选面板内 `source` / `key` 分面的过滤语义。
8. “全选”不得移除当前已选集合中但被分面暂时隐藏的标签；其语义固定为“将当前可见候选全部加入已选集合”。

## 数据读取与刷新约束

1. 顶部标签过滤读取标签时，不得以 `source=meta.annotation` 作为前置条件；默认必须基于统一标签模型读取全部来源标签。
2. 用户点击“包含标签”或“排除标签”按钮并打开面板时，前端必须立即强制刷新当前 root 的标签快照，并从数据库读取最新标签信息。
3. 当访问模式允许人物管理且当前 root 可读取人物复核数据时，打开面板还必须同步刷新当前 root 的人物复核虚拟候选快照（`未归属` / `误检-忽略`）。
4. 面板可先展示缓存候选，再在刷新完成后无感更新；但不得继续停留在仅首次进入 root 时的旧快照。
5. 标签快照刷新后，候选列表、`source` 分面集合与 `key` 分面集合都必须基于最新快照重新计算。
6. 若刷新后当前选中的 `source` 或 `key` 分面值已不再有效，系统必须自动回退到“全部”。

## 候选列表与特殊项语义

1. 顶部标签过滤候选必须按逻辑标签 `key + value` 聚合，不得按 `source` 分裂显示重复候选。
2. 同一文件同一逻辑标签拥有多个来源时，候选计数与文件匹配都只按一个逻辑标签计算。
3. “未标注”语义固定为：当前文件在统一标签模型中不存在任何来源标签。
4. “未标注”是顶部标签过滤的特殊候选项，不参与真实标签来源与标签键的枚举。
5. `人物管理: 未归属` 语义固定为：当前文件至少存在一个 face，且其状态属于 `manual_unassigned | deferred | unassigned`。
6. `人物管理: 误检/忽略` 语义固定为：当前文件至少存在一个 `ignored` face。
7. 两个人物管理候选项属于顶部工具栏虚拟标签，不是持久化逻辑标签，不参与真实标签来源与标签键的枚举，也不得反写到 annotation sidecar 或 `vision.face`。
8. 仅当 `source=全部` 且 `key=全部` 时，所有特殊候选项（`未标注`、`人物管理: 未归属`、`人物管理: 误检/忽略`）才允许显示在候选列表中。
9. 当任一分面生效时，所有特殊候选项都必须隐藏，因为它们不具备可参与 `source` / `key` 分面的真实元数据。

## `source` / `key` 分面规则

1. 顶部标签过滤候选面板必须提供两个显式分面控件：`source` 与 `key`。
2. `source` 分面默认值为“全部来源”；可选值来自当前候选集中 `sources[]` 的去重集合。
3. `key` 分面默认值为“全部 key”；可选值来自当前候选集中 `key` 的去重集合。
4. `source` 与 `key` 分面组合语义固定为 AND。
5. 当选中某个 `source` 时，候选项必须满足 `option.sources` 包含该来源。
6. 当选中某个 `key` 时，候选项必须满足 `option.key` 等于该键。
7. `source` 分面选项列表必须按当前 `key` 选择收窄；`key` 分面选项列表必须按当前 `source` 选择收窄。
8. 分面变化只影响候选面板内显示哪些候选项，不得直接作为文件结果过滤条件。

## 与文件结果过滤的边界

1. 文件结果的实际过滤条件仍固定为：
   - include 已选 `tagKey`
   - exclude 已选 `tagKey`
   - include 匹配模式 `OR/AND`
2. 单独切换 `source` / `key` 分面而未勾选任何标签时，不得改变文件结果集。
3. `source` / `key` 分面不得扩展为工作区级持久过滤条件。

## 验收标准 (AC)

1. `AC-TTF-01` 打开“包含标签”或“排除标签”面板时，系统会立即强制刷新当前 root 标签快照；若数据库标签已变化，面板候选可在当前打开会话内更新到最新结果。
2. `AC-TTF-02` 顶部标签过滤候选默认按 `key + value` 聚合；同一逻辑标签具有多个来源时，列表中仅显示一个候选。
3. `AC-TTF-03` 候选面板内 `source` 与 `key` 分面默认值均为“全部”。
4. `AC-TTF-04` 当用户同时选择某个 `source` 与某个 `key` 时，候选列表按两者交集过滤。
5. `AC-TTF-05` 当任一分面生效时，“未标注”候选项隐藏；清回“全部”后重新显示。
6. `AC-TTF-06` 从 include 面板切换到 exclude 面板，或关闭后重新打开同一面板时，`source` / `key` 分面会重置为“全部”。
7. `AC-TTF-07` 仅调整 `source` / `key` 分面而不勾选标签时，文件结果列表不发生变化。
8. `AC-TTF-08` 点击“全选”后，当前分面可见的全部候选都会进入已选集合；此前已选但当前不可见的标签保持不变。
9. `AC-TTF-09` 在同一 root 中刷新或重进后，顶部标签过滤会恢复上次保存的 include/exclude 与 `OR/AND` 状态；`source` / `key` 分面与面板开关不会恢复。
10. `AC-TTF-10` 当持久化恢复的某个 `tagKey` 已不在当前候选列表中时，系统仍保留该过滤条件并继续按其过滤文件结果，不会在恢复时自动移除。
11. `AC-TTF-11` 在可用的人物管理 root 中打开顶部标签面板后，候选列表会额外显示 `人物管理: 未归属` 与 `人物管理: 误检/忽略`；选择其中任一项后，文件结果按对应 face review bucket 过滤，而不是按真实 annotation sidecar 标签过滤。
